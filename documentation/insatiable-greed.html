Insatiable Greed Transcription Guidelines [DRAFT]

by Michael Hawkins and Imogen Ford

Adapted from the Casebooks Project (Young, Hawkins, Ralley, Cambridge 2011) and the Text Encoding Initiative (TEI) P5 Guidelines.

Introduction

These Guidelines are intended to provide an accessible overview of our transcription policy and tagging conventions. A detailed account of how each element and its attributes are to be used can be found in our Element Reference .

While we aim to minimize jargon, familiarity with basic XML concepts is assumed, such as elements (empty and non-empty), attributes, attribute values, and entities. Understanding how to nest elements is essential. If you're new to XML, we recommend reading a “Gentle Introduction to XML.”

A key tool for editors is the XML comment (<!-- -->). These comments will never be seen on the project's website. They provide a useful way to record internal notes that allow editors to discuss on any issues concerning the text, its transcription or encoding. They can flag problems, explain coding decisions, or be used to raise queries: <!-- TRANSC: Greek 'nu' is actually italic 'v' but encoded as nu - mjh -->, or <!-- TODO: Can you make out this word? - mjh -->.

We try to use standardized headings to make it easier to find, such as:

  • <!--CHECK: verify against the original - ar-->
  • <!--TODO: editorial task - ne-->
  • <!--TRANSC: transcription-related observation or issue - mjh-->

Notes are invaluable during final review because they facilitate targeted searches for unresolved issues. It is useful if you sign the comment with your initials.

Take special care with spacing around elements. XML processing software automatically collapses multiple occurrences of any whitespace characters (i.e. spaces, new lines, and tabs) into a single space. This means you mainly need to watch out for cases where spaces shouldn’t be.

Finally, the project uses an intentionally slimmed-down set of elements from the full TEI schema. That means there’s always a chance we’ll run into something unexpected — and the tags won’t yet exist to support it properly. Don’t hack or force a workaround just to get the desired appearance. It’s always better to encode the transcripts semantically. If you’re unsure how to proceed, get in touch with Mike or Imogen as we can always add support for any missing elements.

Document Structure

Each document is enclosed within a root <TEI> element, containing two main components: <teiHeader> and <text> (in that order).

The <teiHeader> contains all the metadata on the document, such as information about its source, what language(s) it contains, and a record of editorial work.

The transcription is placed within <text>, which itself contains a required <body> element.

<body> must include one or more <div> elements, each representing a distinct section of the text, such as chapters, prefaces, or other logical divisions. The <div> tags may be nested if appropriate.

The standard structure looks like this:


                        <TEI>
                            <teiHeader>
                                <!-- metadata goes here -->
                            </teiHeader>
                            <text>
                                <body>
                                    <div/>
                                    <div/>
                                    <div/>
                                    <!-- any number of <div>s, including nested ones -->
                                </body>
                            </text>
                        </TEI>

The Header (<teiHeader>)

Metadata for each electronic file is recorded in a <teiHeader> element, which has three components: <fileDesc>, <profileDesc>, and <revisionDesc> (in that order):


                        <teiHeader>
                            <fileDesc><!-- info about the electronic file and its source --></fileDesc>
                            <profileDesc><!-- info about the languages in the document --></profileDesc>
                            <revisionDesc><!-- a log of work undertaken by editors on the file --></revisionDesc>
                        </teiHeader>

I. <fileDesc> (req)

<fileDesc> describes the electronic file and its source. It contains the following elements:

  • <titleStmt> contains the document’s title and author
  • <publicationStmt> describes the publisher of the XML document.
  • <notesStmt> (optional) contains one or more <note> elements pertaining to the document as a whole.
  • <sourceDesc> provides information on the source of the text.

<titleStmt> (req)

Contains <title> and <author> for the document's title. Add @xml:lang to either if necessary

<publicationStmt> (req)

<publicationStmt> describes the publisher of the XML document, containing <publisher> (Insatiable Greed Project), <pubPlace> (Cambridge), and an empty <date> element with a machine-readable @when attribute (in yyyy-mm-dd format) containing the publication date. In fact, the <date> is the only element you'll ever need to change in this block:


                            <publicationStmt>
                                <publisher>Insatiable Greed Project</publisher>
                                <pubPlace>Cambridge</pubPlace>
                                <date when="2025-06-17">2025</date>
                            </publicationStmt>

<notesStmt> (opt)

<notesStmt> is only needed if you want to add one or more <note> elements that pertain to the document as a whole. It is not to be used for footnote/endnotes. If you wish to add a footnote/endnote into the transcription, it should be encoded according to the instructions below. In practice, most notes will be brief. If a note grows into multiple paragraphs, consider whether it should be split into separate <note> elements.

<note>Lorem ipsum dolor sit amet, consectetur adipiscing elit. Donec nec leo sapien....</note>

You may include links to related transcriptions using <ref>, as outlined in <ref>'s entry in the Element Reference.

While <notesStmt> is flexible, it's preferable to use more specific semantic tagging where possible.

<sourceDesc> (reg)

<sourceDesc> provides information on the source of the text. Since we're only concerned with printed works, it will contain a <biblStruct> element. The coding of the bibliographical details of the document can be quite intricate, but they thankfully boil down to a handful of various types of work.

II. <profileDesc>

<langUsage> (req)

<langUsage> must contain one or more <language> elements using ISO 639-3 codes, e.g.:

  • eng (English)
  • grc (Classical Greek)
  • lat (Latin)
  • ara (Arabic)

These should be listed in order of prominence within the document. For example:


                            <langUsage>
                                <language ident="grc">Byzantine Greek</language>
                                <language ident="lat">Latin</language>
                            </langUsage>

For handling language changes within the transcription, see <foreign> in the Guidelines.

III. <revisionDesc>

<revisionDesc> (req)

<revisionDesc> is used to record a readable account of all significant actions taken on the XML file. Each <change> includes a @when date (in yyyy-mm-dd format) and a natural-language summary of the update. The person responsible is identified using a <name> element with a unique @xml:id, typically based on their initial and surname (e.g. arussell, mhawkins, nevans). Subsequent mentions of a person already declared anywhere in the file should use @sameAs that points to the @xml:id of the relevant <name> element. For example:


                            <revisionDesc>
                                <change when="2025-10-01">Transcribed by <name xml:id="arussell">Angus Russell</name>.</change>
                                <change when="2025-10-10">Text proofed by <name xml:id="nevans">Nicholas Evans</name>.</change>
                                <change when="2025-10-11">Corrections entered by <name sameAs="#arussell">Angus Russell</name>.</change>
                            </revisionDesc>

Note: You don't need to record every time you modify the file. You just wanto record significant changes, like finishing the transcription, proofing, performing a change in the coding based on revised Transcription Guidelines, or when the document has been released. Proofreading counts as a revision, even if no changes are made, since it updates the file's editorial status.

The Transcription

Introduction

These guidelines are a work in progress, so expect them to be revised quite a lot during the early phases of the project.

Normalisation and Correction

The line between normalisation and correction can be subtle, but the key distinction lies in editorial judgement. When the transcriber or editor believes part of the text is erroneous it should be encoded using a <choice> element, containing both <sic> (the original) and <corr> (the correction).

<choice><sic>civittiis</sic><corr cert="high" resp="#mjhawkins">civitatis</corr></choice>

<corr> reflects what the author would likely have written if the error had been noticed. It may include @cert and @resp attributes to indicate the editor’s level of confidence (high, medium, or low) and who made the correction.

As shown above, both <sic> and <corr> should contain entire words (or numerals), even if only a single character is affected.

Corrections of Redundant or Irrelevant Text

If the author or scribe inadvertently repeats or inserts irrelevant content, use an empty <corr> with a @type with the value noText:

Isabel Carter of <choice><sic>of</sic><corr type="noText"/></choice> 36 yeres

Space Corrections

You may also use <sic> and <corr> to address errors in spacing:

  • <choice><sic>twowords</sic><corr>two words</corr></choice>
  • <choice><sic>a gainst</sic><corr>against</corr></choice>

Content Tagging

Headings

Use <head> to tag headings, whether they belong to the whole document or a specific section. The @rend attribute records the block-level formatting, such as its alignment (centre, flushLineStart, flushLineEnd) and the indentation of the first line (indent0, indent5, indent10). If @rend is omitted, the heading is assumed to be flush with the start of the line (relative to the direction of the text), and the first line will be indented by approximately 5 spaces (equivalent to indent5).

<head> can only appear at the beginning of a <div> or <lg> (line group, typically for verse).

Text that resembles a heading but doesn't introduce a new section should use:

  • <ab type="head"> if it appears outside other block elements (like <p>), or
  • <seg type="head"> if it occurs within a paragraph—though this should be exceedingly rare.

Paragraphs

Prose paragraphs should be tagged using <p>. The @rend attribute captures block-level formatting (i.e. alignment and indentation of the first line). If the paragraph has standard indentation (roughly 3–5 spaces), there’s no need to include an indentation value.

Only record deviations when they seem typographically or semantically significant, such as no indentation, or unusually deep indentation, on the first line of the paragraph. Only use @rend to encode block-level alignment changes (e.g. centre, flushLineEnd); if you need to record text formatting, such as italics, this will have to be done with an inline element as per the instructions in Text formatting with @rend (below).

Multiple formatting instructions may be included in a single @rend value: <p rend="flushLineStart indent0">Lorem ipse dolor zzrill ....</p>

While you may use indent5 to record normal first line indentation, or flushLineStart to indicate that the text is flush to the start of the line ( relative to reading direction), this is not required as these are the default values for all block-level elements (<p>, <list>, <item>, <table>, <cell>, <row>, or <ab>).

Refer to the <p> entry in the Element Set for further examples. If you need to record formatting that isn’t currently supported, please contact Michael Hawkins.

Text Formatting with @rend

The character formatting of text should be described using the @rend attribute on a suitable inline-level element, such as <hi>. For example: this text is <hi rend="italic">italic</hi> and <hi rend="underline">this is underlined</hi>.

Valid values for @rend include:

  • underline
  • doubleUnderline
  • overline
  • italic
  • bold
  • superscript
  • subscript
  • large — slightly larger than normal text
  • larger - even larger
  • largest - the largest size we display
  • small — slightly smaller than normal text
  • smaller- even smaller
  • smallest - the smallest size we display
  • dropCap – for a decorative drop capital at the start of a paragraph
  • smallCaps – for text in Small caps
  • uppercase – for full uppercase text
Small caps vs. Uppercase

Pay close attention to the distinction between smallCaps and uppercase. Although they may look similar, they are typographically distinct:

  • Small Caps is rendered using capital letters that are typically the height of a lowercase letter (technically, the x-height). Initial letters in the words may sometimes be the same size as a standard capital.
  • Uppercase: All letters are full-sized capitals, as in standard capitalisation.
Encoding Small Caps

To encode small caps, use: <hi rend="smallCaps">...</hi>

Be precise with your character input:

  • Smaller capitals should be typed in lowercase
  • Normal-sized capitals should be typed in uppercase

Examples:

  • <hi rend="smallCaps">This string is in Small Caps</hi>
     will look like This string is in Small Caps
  • <hi rend="smallCaps">all smaller caps</hi>
     will look like
Encoding Uppercase

When a word is capitalised normally (e.g. Rome) or includes standard-height abbreviations (e.g. S.P.Q.R), simply enter the characters as they appear, without using @rend.

Only use: <hi rend="uppercase">...</hi> if the text is fully in uppercase for presentational reasons.

When entering text within <hi rend="uppercase">, manually normalise the capitalisation. That means ensuring the text reflects the correct casing as in the examples above.

Foreign Language Text

Any word or passage in a language different from that of the surrounding text must be tagged with <foreign>, and must include an @xml:lang value that matches the @xml:id of a <language> element defined in the <teiHeader>.

Example (Latin within English, with no special formatting):

<foreign xml:lang="lat">Lorem ipsum dolor sit amet, consectetur adipiscing elit</foreign>

If the foreign text is presented with distinctive formatting (e.g. italics), this should be indicated using @rend, with values from the character styling section.

If a foreign passage crosses element boundaries, use nested <foreign> elements within each block, e.g.

... main English text of document followed by <foreign xml:lang="lat" rend="italic">Lorem ipsum dolor sit amet (<foreign xml:lang="gre" rend="roman">φιλεῖ </foreign>), consectetur adipiscing elit</foreign> before the English text resumes ....

Names of People and Places

All person and place names should be encoded using <name> or <placeName>, respectively.

Names within the transcribed text must have a @type with a value of one of the following:

  • group
  • org
  • person
  • uncertain
  • ethnonym

If a name is rendered with a distinctive formatting in the original text (e.g. italics or bold), this should be captured using @rend.

Example:

...lorem ipse <name type="person" rend="italic">Basil</name> dolor zzrill....

Specialised Terms

Use <term> for specialised terms. If the original text displays them with special formatting (e.g. italic), add an appropriate @rend value.

<rs> is used to mark up phrases that indicate the features of the text, such as whether the text indicates that an gift exchange had taken place. Once we start to settle on some values, it will require a @type attribute.

As always, apply @rend for any formatting that appears in the source.

Line breaks

We generally do not record line breaks of within the text. At present, there is only one instance in which we will encode line breaks: when a word with a soft hyphen starts at the very end of one page and finishes on the following page.

In order to decide whether to use encode the line break, it is vital to be aware of the distinction between hard and soft hyphens.

A hyphen that normally occurs in a word, like ‘ill-timed’ is called a hard hyphen. When you encounter a hard hyphen, enter it as you normally would (e.g. 'ill-timed').

A soft hyphen, in constrast, is one that occurs solely for presentational or typographical reasons; it would not occur if the word were in the middle of the line. Soft hyphens appear at the end of lines.

When you encounter a hyphenated word at the end of the last line on a page, determine whether the hyphen is hard or soft. If it's a hard hyphen, enter it as normal. If it's a soft hyphen, do not enter the hyphen. Instead, place <lb rend="hyphen"/> in the hyphen's place. See the example in <pb> below.

Page breaks

Use <pb> to indicate page breaks, including those at the start of a document. It must include both:

  • @n: the page number. This is what appears as the page number in the online view (does not need to be unique).
  • @xml:id: a unique identifier within the document. @xml:id values must start with a letter, it will be necessary to insert add the prefix p (for page) to the page number: e.g. p12 (for page 12). If page numbers are not unique within the document, it will be necessary to append a -2, -3 to subsequent appearances of the same page within that document, e.g. p20 (for the first occurence of page 20; p20-2 (for the second occurence); p20-3 (for the third), etc.

When the page break occurs between words, be sure to include at least one space on either side of the <pb>, e.g. ... the <pb xml:id="p27" n="27"/> advowson ....

If a word is split by a page break, insert <pb> in the middle of the word with no spaces on either side. If the word contains soft hyphen, you would place <lb type="hyphenated"/> immediately before the <pb>.). For example, ... the ad<lb rend="hyphen"/><pb xml:id="p27" n="27"/>vowson...

Section Markers

Use <milestone> to record significant section markers, such as a canonical paragraph number or section number, associated with that section of the text. It requires:

  • @n contains the value of the marker that we'd want to see online (eg. 1337)
  • @xml:id contains a version of the marker so that it's possible to link to it. @xml:id must begin with a letter, so if the marker was 1337, you'd have to insert a suitable prefix, para (for paragraph), sect for section. Thus, paragraph 1337 would have an @xml:id of para1337; section 1337 would have sect1337.
  • @type (currently only allows one value genericMarker, but more can be added if need be.

Notes

Notes added by the XML Editor

Use <note type="editorial"> to record editorial notes added by the XML editor. These notes should be placed within the text where you want their indicators to be placed.

...lorem ipse dolor zrill.<note type="editorial">In her <hi rend="italic">Proin pellentesque maximus blandit</hi>, Bloggs claimed that .....</note>

If the note contains more than one paragraph, use <p> for each paragraph.

No other attributes should be used on editorial notes.

Editorial notes will be automatically numbered online. Their numbered indicators will appear in the text where the <note> element was; the footnotes themselves will be numbered and placed at the bottom of the page.

Notes in the original source

It's unlikely that we would want to transcribe a text and its notes. If we did, use <note> (without @type) to record any marginal notes, footnotes or endnotes transcribed from the original source.

The note's indicator, such as an obelus (†), asterisk (*), or superscript character (1, 2, 3), would be recorded using the @n attribute. This is the character that will be used online for that note.

The physical placement of the note in the source document may be recorded using the @place attribute. Accepted values include:

  • infralinear
  • inline
  • interlinear
  • lineStart
  • lineEnd
  • marginLeft
  • marginRight
  • pageTop
  • pageBottom

These @place values are currently constrained to a predefined set but may be expanded in future if necessary.

The value must point directly to one or more XML elements or other resources by means of one or more URIs, separated by whitespace. If more than one is supplied the implication is that the name identifies several distinct entities.

Example

In this contrived example, a canonical reference to the same organisation is provided in four different ways.


      <author n="1">
        <name ref="http://nzetc.victoria.ac.nz/tm/scholarly/name-427308.html" type="organisation">New Zealand Parliament, Legislative Council</name>
      </author>
       
      <author n="2">
        <name ref="nzvn:427308" type="organisation">New Zealand Parliament, Legislative Council</name>
      </author>
          
      <author n="3">
        <name ref="./named_entities.xml#o427308" type="organisation">New Zealand Parliament, Legislative Council</name>
      </author>
       
      <author n="4">
        <name key="name-427308" type="organisation">New Zealand Parliament, Legislative Council</name>
      </author>

The first presumes the availability of an internet connection and a processor that can resolve a URI (most can). The second requires, in addition, a <prefixDef> that declares how the nzvm prefix should be interpreted. The third does not require an internet connection, but does require that a file named named_entities.xml be in the same directory as the TEI document. The fourth requires that an entire external system for key resolution be available.

Note

The @key attribute is more flexible and general-purpose, but its use in interchange requires that documentation about how the key is to be resolved be sent to the recipient of the TEI document. In contrast values of the @ref attribute are resolved using the widely accepted protocols for a URI, and thus less documentation, if any, is likely required by the recipient in data interchange.

These guidelines provide no semantic basis or suggested precedence when both @key and @ref are provided. For this reason simultaneous use of both is not recommended unless documentation explaining the use is provided, probably in an ODD customizaiton, for interchange.

Declaration

Note

The values for this attribute are language ‘tags’ as defined in BCP 47. Currently BCP 47 comprises RFC 5646 and RFC 4647; over time, other IETF documents may succeed these as the best current practice.

A ‘language tag’, per BCP 47, is assembled from a sequence of components or subtags separated by the hyphen character (-, U+002D). The tag is made of the following subtags, in the following order. Every subtag except the first is optional. If present, each occurs only once, except the fourth and fifth components (variant and extension), which are repeatable.

language
The IANA-registered code for the language. This is almost always the same as the ISO 639 2-letter language code if there is one. The list of available registered language subtags can be found at https://www.iana.org/assignments/language-subtag-registry. It is recommended that this code be written in lower case.
script
The ISO 15924 code for the script. These codes consist of 4 letters, and it is recommended they be written with an initial capital, the other three letters in lower case. The canonical list of codes is maintained by the Unicode Consortium, and is available at https://unicode.org/iso15924/iso15924-codes.html. The IETF recommends this code be omitted unless it is necessary to make a distinction you need.
region
Either an ISO 3166 country code or a UN M.49 region code that is registered with IANA (not all such codes are registered, e.g. UN codes for economic groupings or codes for countries for which there is already an ISO 3166 2-letter code are not registered). The former consist of 2 letters, and it is recommended they be written in upper case; the list of codes can be searched or browsed at https://www.iso.org/obp/ui/#search/code/. The latter consist of 3 digits; the list of codes can be found at http://unstats.un.org/unsd/methods/m49/m49.htm.
variant
An IANA-registered variation. These codes ‘are used to indicate additional, well-recognized variations that define a language or its dialects that are not covered by other available subtags’.
extension
An extension has the format of a single letter followed by a hyphen followed by additional subtags. There are currently only two extensions in use. Extension T indicates that the content was transformed. For example en-t-it could be used for content in English that was translated from Italian. Extension T is described in the informational RFC 6497. Extension U can be used to embed a variety of locale attributes. It is described in the informational RFC 6067.
private use
An extension that uses the initial subtag of the single letter x (i.e., starts with x-) has no meaning except as negotiated among the parties involved. These should be used with great care, since they interfere with the interoperability that use of RFC 4646 is intended to promote. In order for a document that makes use of these subtags to be TEI-conformant, a corresponding <language> element must be present in the TEI header.

There are two exceptions to the above format. First, there are language tags in the IANA registry that do not match the above syntax, but are present because they have been ‘grandfathered’ from previous specifications.

Second, an entire language tag can consist of only a private use subtag. These tags start with x-, and do not need to follow any further rules established by the IETF and endorsed by these Guidelines. Like all language tags that make use of private use subtags, the language in question must be documented in a corresponding <language> element in the TEI header.
